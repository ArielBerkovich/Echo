# AI/operator installation runbook

Use this file when installing or upgrading the Echo chart. Do not edit chart
templates for environment-specific configuration; change values only.

## Safety rules

1. Confirm the current Kubernetes context and target namespace before changing
   the cluster.
2. Never print Secret data, generated passwords, registry passwords, or pull
   Secret contents. Report only Secret names and required keys.
3. Do not grant `anyuid`, privileged, or custom SCC access. The OpenShift
   profile is intended for the standard `restricted-v2` SCC.
4. Do not run `helm dependency update` in a disconnected environment. The
   MongoDB and MinIO chart archives are already under `charts/`.
5. Use immutable application and dependency tags or digests. Do not promote
   `latest` into an air-gapped environment.
6. On upgrades, preserve the existing release Secrets. Generated credentials
   must continue to match data on retained MongoDB and MinIO PVCs. The chart
   retains both PVCs during `helm uninstall`; do not delete them unless the
   stored application data should also be removed.

## Preflight

Run read-only checks first:

```bash
kubectl config current-context
kubectl cluster-info
helm version
helm lint ./helm/echo
```

For OpenShift, also verify:

```bash
oc whoami
oc api-resources --api-group=route.openshift.io
oc get storageclass
```

Before an OpenShift installation, edit `values-openshift.yaml` and replace all
example values:

- `artifactory.example.com`
- every `docker-local/...` repository path as required by Artifactory
- Echo, MongoDB, MinIO, and MinIO client tags/digests
- `openshift-storage-class`
- `echo.apps.example.com`

Render and inspect the result before installation:

```bash
helm template echo ./helm/echo \
  --namespace echo \
  --values ./helm/echo/values-openshift.yaml > /tmp/echo-rendered.yaml

grep 'image:' /tmp/echo-rendered.yaml
```

Every rendered image must point to the disconnected registry. There should be
no `docker.io`, `quay.io`, or other internet registry references. There should
also be no rendered `runAsUser`, `runAsGroup`, or `fsGroup` fields in the
OpenShift profile.

## Local Kubernetes installation

The default values use fixed non-root identities suitable for ordinary
Kubernetes:

```bash
helm upgrade --install echo ./helm/echo \
  --namespace echo \
  --create-namespace \
  --wait \
  --timeout 10m
```

For locally built images, import/push immutable tags and override them:

```bash
helm upgrade --install echo ./helm/echo \
  --namespace echo \
  --create-namespace \
  --set server.image.tag='<immutable-tag>' \
  --set client.image.tag='<immutable-tag>' \
  --wait \
  --timeout 10m
```

## Disconnected OpenShift installation

All runtime images must already exist in Artifactory. Create a read-only pull
Secret in the target project; do not put registry credentials in values files:

```bash
oc new-project echo

oc create secret docker-registry artifactory-pull \
  --docker-server='<artifactory-host>' \
  --docker-username='<read-only-user>' \
  --docker-password='<password>'
```

If Artifactory uses a private CA, an OpenShift cluster administrator must add
it to the cluster image configuration. A pull Secret does not configure CA
trust.

Install only after the rendered-manifest checks pass:

```bash
helm upgrade --install echo ./helm/echo \
  --namespace echo \
  --values ./helm/echo/values-openshift.yaml \
  --wait \
  --timeout 15m
```

Authentication credentials for Echo, MongoDB, and MinIO are generated by their
charts when left empty. Installation does not require internet access for this
generation.

## Verification

```bash
kubectl get pods,pvc -n echo
kubectl rollout status deployment/echo-client -n echo --timeout=180s
kubectl rollout status deployment/echo-server -n echo --timeout=180s
kubectl rollout status deployment/echo-minio -n echo --timeout=180s
kubectl rollout status statefulset/echo-mongodb -n echo --timeout=300s
```

On OpenShift:

```bash
oc get route echo-client -n echo
oc get pods -n echo \
  -o custom-columns='NAME:.metadata.name,UID:.spec.securityContext.runAsUser,SCC:.metadata.annotations.openshift\.io/scc'
```

The pods should be ready, PVCs bound, the Route admitted, and the SCC should be
`restricted-v2` or the cluster's equivalent restricted SCC.

For local access without Ingress or Route:

```bash
kubectl port-forward -n echo service/echo-client 8091:80
```

Then verify `http://127.0.0.1:8091/` and `/api/health` through the client proxy.

## Failure handling

Gather diagnostics without exposing Secret values:

```bash
kubectl get pods -n echo
kubectl get events -n echo --sort-by=.lastTimestamp
kubectl describe pod -n echo '<failing-pod>'
kubectl logs -n echo '<failing-pod>' --all-containers --tail=200
helm status echo -n echo
```

Common causes:

- `ImagePullBackOff`: wrong Artifactory path, missing pull Secret, or untrusted CA.
- SCC rejection: a fixed UID/GID remains enabled, usually in MinIO overrides.
- Pending PVC: the configured StorageClass is absent or cannot provision RWO
  storage.
- Route unavailable: wrong host/domain or the Route was not admitted.
- MongoDB authentication failure after reinstall with chart versions before
  0.4.2: retained PVCs may be paired with newly generated credentials. Recover
  the original Secret instead of generating a new password for existing data,
  then upgrade to a chart version that preserves existing Secret values.

Do not uninstall a production release merely to troubleshoot it. If an upgrade
fails and the previous revision was healthy, inspect `helm history echo -n echo`
and use an explicit `helm rollback` revision after confirming with the operator.
